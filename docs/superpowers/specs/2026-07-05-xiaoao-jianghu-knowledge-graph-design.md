# 《笑傲江湖》知识图谱教学平台设计

## 1. 背景与目标

本项目以《笑傲江湖》为内置示范语料，建设一个兼顾业务人员与开发者的知识图谱教学网站。用户先通过人物、门派、武学与事件的直观关系理解知识图谱的价值，再逐层查看本体、图查询、来源证据和抽取过程。

平台同时支持用户上传 TXT 小说，通过 OpenAI 兼容 API 或本地 Ollama 在线构建隔离的小说知识图谱。首版覆盖教学、探索、问答和构建闭环，不定位为通用企业知识图谱治理平台。

### 1.1 成功标准

- 新用户能沿默认导览理解实体、关系、属性、本体、实例和图查询的区别。
- 《笑傲江湖》图谱支持搜索、类型筛选、邻居展开、路径查找、剧情时间线和原文证据定位。
- 自然语言问答同时返回自然语言答案、图路径、参数化 Cypher、命中事实和原文证据。
- 用户上传 TXT 后可观察完整构建进度；中断或局部失败后可以续跑。
- OpenAI 兼容 API 与 Ollama 通过同一模型接口接入，不让前端感知供应商差异。
- 关键本体规则、API、任务状态、图查询和核心用户路径具备自动测试。

### 1.2 首版不包含

- 多人协作审核与复杂权限体系。
- 用户注册、登录和计费。
- PDF、EPUB、DOCX 等任意文件格式。
- 自动训练或微调模型。
- Neo4j 集群、任务队列集群和生产级多租户部署。

## 2. 用户与体验原则

平台面向两类用户：希望理解概念与用途的业务人员，以及希望查看数据模型和实现细节的开发者。

每项功能采用双层呈现：默认业务视图使用人物卡、关系路径、时间线和示例问题解释价值；可展开的技术面板展示节点标签、关系类型、Cypher、JSON、置信度、来源片段、模型和抽取批次。

默认教学路径为：

1. 导览：用一个三元组解释实体、关系和属性。
2. 本体：对照 TBox 与 ABox，理解概念定义和实例事实。
3. 图谱：搜索、筛选、展开邻居并查找人物间路径。
4. 问答：查看答案如何由图查询和原文证据共同支持。
5. 故事线：按章节观察人物关系与组织隶属如何变化。
6. 构建：上传小说并观察从文本到图谱的全过程。

## 3. 总体架构

系统分为六个边界清晰的组件：

### 3.1 Web 前端

使用 React 与 TypeScript。负责教学导览、图谱可视化、本体浏览、问答、上传和任务进度。前端只调用后端 API，不直接访问 Neo4j 或模型服务。

### 3.2 API 服务

使用 FastAPI。负责项目管理、图查询、问答编排、模型配置、文件上传、任务控制和 SSE 进度推送。所有 Cypher 使用参数化查询或受控模板，客户端不能提交任意 Cypher。

### 3.3 抽取任务执行器

任务执行器与 API 进程逻辑解耦，首版可作为独立进程运行。它负责章节切分、片段抽取、实体消歧、事实合并、本体校验和批量入图。任务与片段状态持久化，因此进程重启后可续跑。

### 3.4 图数据库

Neo4j 存储领域实体、关系、章节、证据片段、抽取批次和审核状态。不同小说使用 `project_id` 隔离；所有查询必须带项目范围，避免跨项目混图。

### 3.5 模型适配层

模型适配层向抽取与问答提供统一接口，支持 OpenAI 兼容 API 和 Ollama。配置包括供应商、基础地址、模型名、密钥、超时、并发数和上下文限制。密钥仅保存在后端运行环境或本地安全配置中，不返回前端。

### 3.6 任务元数据与文件存储

首版本地部署使用 SQLite 保存项目、任务、片段状态和模型配置引用，上传文本保存在按项目隔离的受控文件目录。SQLite 与 Neo4j 职责分离：前者管理流程状态，后者管理可查询的知识。密钥只通过环境变量或运行时输入提供，不写入 SQLite。该边界允许后续将 SQLite 和本地文件替换为 PostgreSQL 与对象存储，而不改变抽取及查询接口。

## 4. 领域本体

### 4.1 核心类

- `Person`：人物。
- `Organization`：组织；子类包括 `Sect`、`Clan`、`EscortAgency` 和 `PoliticalForce`。
- `MartialArt`：武学；子类包括 `Swordplay`、`InternalSkill`、`PalmTechnique`、`Qinggong` 和 `MusicScore`。
- `Event`：事件；`TeachingEvent` 是表达武学传授的子类。
- `Place`：地点。
- `Artifact`：兵器、秘籍、信物等物品。
- `Chapter`：章节。
- `Evidence`：可定位的原文证据片段。

### 4.2 核心关系

- `MEMBER_OF`：人物隶属于组织。
- `MASTER_OF`：人物是另一人物的师父。
- `KIN_OF`：人物之间的亲属关系，附带亲属类型。
- `ALLY_OF`、`ENEMY_OF`：盟友与敌对关系。
- `KNOWS`：人物掌握武学。
- `TEACHER`、`STUDENT`、`SUBJECT`：将 `TeachingEvent` 分别关联到传授者、学习者与武学，准确表达三元事实。
- `HOLDS`：人物持有物品。
- `PARTICIPATES_IN`：人物或组织参与事件。
- `OCCURS_AT`：事件发生于地点。
- `EVIDENCED_BY`：实体或事实由证据片段支持。

### 4.3 约束

- 类型约束：例如 `KNOWS` 的主体必须是 `Person`，客体必须是 `MartialArt`。
- 方向约束：师父与弟子使用确定方向，查询层可提供逆向显示，不重复存储矛盾关系。
- 时态约束：隶属、盟友、敌对、持有等关系允许带 `from_chapter` 和 `to_chapter`。
- 证据约束：进入确认图谱的事实至少关联一个可定位证据。
- 项目约束：关系两端、证据和章节必须属于同一项目。

## 5. 图数据与来源追踪

每个实体保存稳定标识、规范名称、别名、类型、简介、首次出现章节、项目标识和审核状态。每条候选事实保存关系类型、方向、时间范围、置信度、抽取模型、抽取批次和审核状态。

事实的来源追踪不能只保存在关系属性中。系统使用事实记录或关系标识将一条事实关联到多个 `Evidence` 节点，从而支持跨章节重复佐证、证据撤销和质量审计。证据保存章节、字符偏移、短片段和文本指纹，不复制无边界的长篇原文。

实体消歧以“项目内规范实体”为目标，综合别名、同现人物、所属组织、上下文描述和时间位置。无法可靠合并的实体保持为待审核候选，禁止仅凭同名强制合并。

## 6. 在线抽取数据流

1. 上传：校验 TXT 文件大小、编码和空内容，生成隔离项目。
2. 切分：识别章节；章节过长时按语义段落切成带重叠窗口的片段。
3. 抽取：模型根据本体输出受 JSON Schema 约束的实体、关系和证据偏移。
4. 局部归一化：规范名称、关系方向和枚举值，拒绝不在本体中的类型。
5. 实体消歧：先做确定性别名匹配，再对模糊候选调用模型判定。
6. 跨章节合并：合并重复事实，累计证据和置信信息。
7. 本体校验：检查类型、方向、项目范围、时间范围和证据完整性。
8. 入图：使用幂等键批量写入 Neo4j；重复执行不会产生重复节点或关系。
9. 报告：生成实体数、关系数、类型分布、低置信候选、失败片段和成本统计。

任务状态包括 `queued`、`splitting`、`extracting`、`resolving`、`validating`、`writing`、`completed`、`paused`、`failed` 和 `cancelled`。片段级状态单独保存，以支持续跑和局部重试。

## 7. 图谱探索与问答

### 7.1 探索能力

- 按实体名和别名搜索。
- 按人物、组织、武学、事件、地点和物品筛选。
- 展开指定深度的邻居，并限制节点数量防止画布失控。
- 查找两实体之间的受限最短路径。
- 选择章节范围，观察关系随剧情阶段变化。
- 点击节点或关系查看属性、证据、来源章节和技术表示。

### 7.2 问答流程

1. 对问题进行项目范围确认和意图分类。
2. 从受控查询模板中选择查询类型并抽取参数。
3. 参数化执行 Cypher，限制返回节点数、路径深度和执行时间。
4. 基于查询结果和证据片段生成回答。
5. 返回答案、图路径、Cypher 模板及参数、命中事实和证据定位。

图查询无结果时，系统明确说明“图谱中暂无足够事实”，不允许模型用小说常识补写答案。技术视图显示查询过程，但密钥、内部系统提示和完整模型响应不暴露给客户端。

## 8. API 边界

首版 API 按职责分组：

- `/api/projects`：创建、列出、查看和删除小说项目。
- `/api/projects/{id}/upload`：上传 TXT 并创建构建任务。
- `/api/jobs/{id}`：查看、暂停、继续、取消和重试任务。
- `/api/jobs/{id}/events`：通过 SSE 推送阶段、片段进度和错误摘要。
- `/api/ontology`：返回本体类、关系和约束的只读描述。
- `/api/graph/search`：实体搜索与类型筛选。
- `/api/graph/neighborhood`：受限邻居展开。
- `/api/graph/path`：受限路径查询。
- `/api/entities/{id}`：实体详情、关系和证据。
- `/api/ask`：执行可解释图谱问答。
- `/api/settings/models/test`：测试用户提供的模型配置，不返回密钥。

具体请求、响应字段和错误码在实施计划阶段形成独立接口契约，避免页面组件依赖数据库内部结构。

## 9. 错误处理与恢复

- 模型超时、限流和暂时性网络错误采用带抖动的指数退避，并按片段重试。
- 模型结构化输出无效时保存脱敏后的原始响应与错误原因，使用修复提示重试；超过上限后标记该片段失败。
- API 或执行器重启后，从最近成功片段继续，不重复处理已完成片段。
- 低置信、无证据或违反本体的事实进入待审核区，不写入确认图谱。
- Neo4j 不可用时任务转为暂停或可重试失败，并显示可操作的恢复信息。
- 用户取消任务后停止领取新片段；已开始的模型请求完成后丢弃或安全记录结果。
- 文件格式、大小、编码和模型配置错误在任务启动前返回明确错误，避免产生半成品项目。

## 10. 安全与资源控制

- 上传文件限制类型和大小，使用服务端生成的文件名，禁止路径穿越。
- 模型密钥不写入日志、任务事件、图数据库或前端持久化状态。
- 所有图查询使用允许列表、参数化模板、深度上限、结果上限和超时。
- 模型输入视为不可信文本；抽取结果必须通过 Schema 和本体双重校验。
- 单项目限制并发片段数和重试次数，并向用户显示预计调用量与实际统计。
- 删除项目时清除其图数据、任务元数据和上传文本；失败删除可以安全重试。

## 11. 测试策略

### 11.1 单元测试

- 章节识别、片段切分与偏移映射。
- 本体类型、方向、时间和证据约束。
- 模型适配器的统一输出和错误映射。
- 实体别名归一化、候选匹配和幂等键生成。
- 任务状态转换与重试决策。

### 11.2 集成测试

- 使用固定模型响应完成从 TXT 到 Neo4j 的全流水线。
- 对 OpenAI 兼容接口和 Ollama 使用契约测试，不依赖真实付费调用。
- 验证项目隔离、幂等入图、失败续跑和删除清理。
- 验证受控 Cypher 模板的结果上限、深度限制和无结果行为。

### 11.3 前端与端到端测试

- 教学导览完整可用，技术面板可以展开。
- 图谱搜索、筛选、邻居展开、路径查找和证据定位可用。
- 上传后能看到实时进度，失败任务能重试并继续。
- 问答同时显示答案、路径、查询说明和证据，无结果时不产生无依据回答。
- 大图数据下交互保持可用，画布不会因无限展开而失控。

## 12. 分阶段交付

### 阶段一：可信的《笑傲江湖》教学演示

建立本体、项目骨架、Neo4j 数据模型和教学界面；先通过可重复的离线流水线生成《笑傲江湖》图谱，并对核心人物、门派、武学和事件进行抽样校验。该阶段必须完成探索、证据和问答闭环。

### 阶段二：在线通用构建

加入上传、模型配置、异步任务、进度、重试、质量报告和项目隔离。在线构建复用阶段一的本体和抽取组件，不另写一套数据逻辑。

### 阶段三：质量改进

根据真实抽取结果优化章节识别、实体消歧、关系时态和审核体验。只有在质量数据证明必要时才增加更复杂的队列、缓存或人工协作能力。

## 13. 设计决策摘要

- 选择 React、FastAPI 与 Neo4j，以真实展示图查询而非仅绘制关系图。
- 选择可插拔模型适配层，同时支持 OpenAI 兼容 API 和 Ollama。
- 将本体、图谱实例和原文证据分别建模，确保教学清晰且事实可追溯。
- 将在线构建设计为持久化异步任务，适应长篇小说和模型限流。
- 先交付可信内置演示，再开放通用上传，降低“平台完整但样例不可信”的风险。
