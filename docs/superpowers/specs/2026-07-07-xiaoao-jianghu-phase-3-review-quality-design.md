# 《笑傲江湖》知识图谱阶段三：审核工作台与质量改进设计

## 1. 目标

阶段三在阶段二“在线通用构建”的基础上增加人工审核与质量修正闭环。目标不是重写抽取流水线，而是让模型抽取后的质量问题可见、可解释、可修正，并让审核后的图谱成为默认查询结果。

阶段三第一版交付“审核工作台 + 质量仪表盘”：

- 生成项目级待审核项；
- 支持接受或拒绝事实；
- 支持合并重复实体；
- 支持从实体中拆出别名，并迁移相关事实和证据；
- 默认图谱和问答读取审核后有效结果；
- 保存可追溯审计日志；
- 展示基础质量指标和审核前后变化。

## 2. 非目标

阶段三第一版不包含：

- 正式登录、权限和多人审批；
- PostgreSQL、Redis、外部队列或分布式 Worker；
- 拖拽式复杂实体拆分工作台；
- 自动重抽取或主动调用模型重新修复；
- 删除原始证据或原始抽取记录；
- 完整课程化教学页面。

数据模型预留 reviewer 字段，但 UI 按单机单用户处理，第一版统一写入 `local_reviewer`。

## 3. 总体架构

阶段三新增审核层，保留阶段二现有边界：

- FastAPI 继续作为唯一后端入口；
- SQLite 保存审核队列、审核动作、质量快照和审计日志；
- Neo4j 保存项目图谱、实体/事实审核状态、合并指针和默认查询所需状态；
- React 新增“审核”主导航入口，提供审核工作台和质量仪表盘。

数据流：

```text
阶段二构建完成
  → 生成原始候选图谱与质量报告
  → 阶段三规则扫描生成 ReviewItem
  → 审核员接受/拒绝/合并/拆出
  → 写入 ReviewAction 审计日志
  → 更新实体/事实审核状态
  → 图谱/问答默认读取审核后视图
  → 质量仪表盘展示审核前后变化
```

核心原则：

- 自动抽取、人工审核、最终展示三层边界清楚；
- 审核后图谱是默认视图；
- 原始抽取记录和证据保留，不因审核动作被删除；
- 所有操作必须限定 `project_id`；
- 所有写操作必须幂等。

## 4. 审核对象与状态模型

### 4.1 ReviewItem 类型

阶段三第一版支持三类审核项。

#### 事实审核项

目标是判断一条事实是否应进入默认图谱和问答。

来源：

- 低置信；
- 证据缺失；
- 证据 quote 与源片段不匹配；
- 本体冲突；
- 用户从图谱页手动加入；
- 模型建议。

动作：

- `accept_fact`：事实进入默认图谱和问答；
- `reject_fact`：事实从默认图谱和问答排除，但保留审计记录和证据；
- `request_more_evidence`：保留为状态，不触发自动重抽取。

#### 重复实体候选

目标是判断两个实体是否应合并。

来源：

- 同名；
- 别名相似；
- 同章节高频共现；
- 模型建议；
- 用户手动加入。

动作：

- `merge_entities`：把 source 实体并入 target 实体，事实和证据重定向到 target。

合并不物理删除 source。source 标记 `merged_into=target_id`，用于审计、回溯和后续回滚能力。

#### 别名拆出候选

目标是处理某个别名或一组事实不应属于当前实体的情况。

动作：

- `split_alias`：从实体上移出某个 alias，创建新实体或绑定到已有实体；
- 迁移与该 alias 强相关的事实和证据。

第一版不做复杂拖拽拆分，只做“别名 + 关联事实”级别拆出。

### 4.2 状态

`ReviewItem` 状态：

- `OPEN`
- `RESOLVED`
- `DISMISSED`

`Fact` 状态：

- `PENDING`
- `ACCEPTED`
- `REJECTED`

`Entity` 状态：

- `PENDING`
- `ACCEPTED`
- `MERGED`
- `SPLIT_SOURCE`

`ReviewAction` 是 append-only 记录，不更新、不删除。

## 5. 后端 API 与数据边界

阶段三新增 `review` 模块，保持和现有 `projects`、`jobs`、`extraction`、`graph` 边界一致。

建议 API：

- `GET /api/projects/{project_id}/review/summary`  
  返回待审数量、事实状态统计、实体状态统计、证据覆盖率、审核前后变化。

- `GET /api/projects/{project_id}/review/items?type=&status=&limit=&cursor=`  
  分页返回审核队列。每个 item 包含原因、候选对象、证据摘要和来源：`rule`、`model`、`manual`。

- `POST /api/projects/{project_id}/review/items`  
  手动加入审核项。第一版用于从图谱页把事实或实体送入审核队列。

- `POST /api/projects/{project_id}/review/items/{item_id}/actions`  
  统一提交审核动作。后端按 action type 分派到对应服务。

- `GET /api/projects/{project_id}/review/audit?entity_id=&fact_id=&limit=&cursor=`  
  查询审计日志。

- `POST /api/projects/{project_id}/review/scan`  
  对已构建项目重新扫描，生成或刷新系统规则审核项。该接口幂等，不重复创建相同待审项。

数据边界：

- SQLite 保存 `review_items`、`review_actions`、`quality_snapshots`；
- Neo4j 保存实体和事实的当前审核状态、合并指针、默认图谱查询所需字段；
- 审计日志以 SQLite 为准；
- Neo4j 只保存当前有效状态；
- 图谱页和问答页不需要读取 SQLite 才能过滤已拒绝事实。

错误处理：

- 已解决 item 再次提交动作返回当前结果，不重复写操作；
- 合并不存在、跨项目、类型不匹配、本体冲突，返回稳定错误码；
- 拆出别名时如果 alias 不存在或事实不属于该实体，拒绝操作并保留 item 为 `OPEN`。

## 6. 前端审核工作台

阶段三新增主导航入口：`审核`。它是构建后的质量治理入口，不隐藏在构建页内。

页面结构：

### 6.1 顶部质量概览

展示当前项目的质量状态：

- 待审核项总数；
- 待审事实、待审实体、别名拆出候选；
- 已接受事实数；
- 已拒绝事实数；
- 证据覆盖率；
- 实体合并数；
- 事实减少数；
- 事实接受率。

### 6.2 审核队列

左侧列表按类型和严重度排序：

1. 本体冲突；
2. 无证据或证据不匹配；
3. 低置信事实；
4. 重复实体候选；
5. 别名拆出候选；
6. 手动加入项。

支持按类型、状态和来源筛选。

### 6.3 审核详情面板

右侧展示当前 item：

- 问题原因；
- 候选事实或候选实体；
- 证据片段、章节、偏移、来源模型；
- 当前状态；
- 历史操作；
- 可执行动作按钮：接受、拒绝、合并、拆出、忽略。

### 6.4 审计日志抽屉

用户可以查看某个实体或事实的所有审核动作，说明谁在什么时候因为什么改了它。第一版 reviewer 固定显示 `local_reviewer`。

### 6.5 图谱页联动

图谱页实体和事实详情处增加“加入审核”入口。默认图谱只展示审核后有效数据；被拒绝事实从图谱和问答中消失。

第一版不做复杂可视化拖拽。实体合并和别名拆出使用表单和确认对话框完成。

## 7. 质量规则

阶段三第一版的规则扫描保持确定性，不依赖模型即可跑通。

系统规则生成的 `ReviewItem` 包括：

- `LOW_CONFIDENCE_FACT`：事实置信度低于阈值；
- `MISSING_EVIDENCE`：事实没有可追溯证据；
- `EVIDENCE_MISMATCH`：证据 quote 与源片段不匹配；
- `ONTOLOGY_VIOLATION`：关系类型不允许连接当前实体类型；
- `POSSIBLE_DUPLICATE_ENTITY`：同名、别名重叠或规范名高度相似；
- `ALIAS_SPLIT_CANDIDATE`：某个 alias 只支撑少量独立事实，且与主实体上下文弱相关；
- `MANUAL_REVIEW`：用户从图谱页手动加入。

模型建议可以作为 `source=model` 的审核项进入队列，但第一版验收不依赖模型建议质量。

## 8. 质量仪表盘指标

第一版指标：

- `open_review_items`
- `accepted_facts`
- `rejected_facts`
- `pending_facts`
- `merged_entities`
- `split_aliases`
- `evidence_coverage`
- `review_completion_rate`
- `graph_fact_delta_before_after_review`

指标口径：

- 审核前基线来自扫描时的项目图谱状态；
- 审核后状态来自 Neo4j 当前有效图谱；
- 审计趋势来自 SQLite 的 `review_actions`；
- 指标必须按 `project_id` 聚合。

## 9. 验收标准

阶段三第一版完成后必须满足：

- 一个固定测试项目能生成至少 3 类待审项；
- 接受事实后，该事实出现在默认图谱和问答中；
- 拒绝事实后，该事实从默认图谱和问答中消失，但审计日志仍可查；
- 合并实体后，图谱只显示目标实体，source 标记为 `MERGED`，不物理删除；
- 拆出 alias 后，新实体或目标实体接收 alias 及相关事实；
- 重复执行 scan 不产生重复审核项；
- 重复提交同一 action 不产生重复审计动作；
- 阶段一教学演示保持可运行；
- 阶段二在线构建流程保持可运行；
- `make verify` 继续作为全量验收入口。

## 10. 风险与约束

- 审核后图谱作为默认视图，会影响现有图谱和问答查询。查询层必须统一过滤 `REJECTED` 和 `MERGED` 状态，避免不同页面结果不一致。
- 实体合并和别名拆出涉及 Neo4j 批量更新，必须以项目范围和幂等键保护。
- SQLite 审计日志和 Neo4j 当前状态可能出现部分失败。写操作应先校验，再在服务层按明确顺序执行，并在失败时返回稳定错误码。
- 第一版不引入账户系统，因此 reviewer 只用于未来兼容，不能作为安全边界。
- 模型建议只作为审核项来源，不作为自动决策来源。

## 11. 实施切片建议

后续实施计划应按可验证切片推进：

1. 审核数据模型、仓储和幂等 scan；
2. 事实接受/拒绝动作与默认图谱过滤；
3. 审计日志与质量 summary；
4. 重复实体候选和合并动作；
5. 别名拆出动作；
6. 审核工作台 UI；
7. 图谱页加入审核与问答过滤回归；
8. E2E、README、版本发布准备。
